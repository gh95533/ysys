# greenplum truncate vacuum vacuum full 

**document support**

ysys

**date**

2020-09-10

**label**

greenplum,truncate,vacuum,vacuum full

**level**

middle



## background

​	最近一段时间,greenplum数据库有些表经常做update导致数据库冗余很高,之后数据库因为空间不顾就给挂了，在这里就讨论一下中间表(就是存储过程中中间的表)是否可以update后truncate掉，是否导致数据库冗余。



## operation

​	在postgresql数据库中得知了数据库truncate table 后等于重建了该表，那么greenplum 是否也是这个样子的。



### case one :update vacuum 

```
gpdw=# create table test001(id int4);
NOTICE:  Table doesn't have 'DISTRIBUTED BY' clause -- Using column named 'id' as the Greenplum Database data distribution key for this table.
HINT:  The 'DISTRIBUTED BY' clause determines the distribution of data. Make sure column(s) chosen are the optimal data distribution key to minimize skew.
CREATE TABLE
gpdw=# insert into test001 select generate_series(1,1000000);
INSERT 0 1000000
gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 117 MB
(1 row)

gpdw=# update test001 set id = id + 100;
UPDATE 1000000
gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 151 MB
(1 row)

gpdw=# vacuum test001;
VACUUM
gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 152 MB
(1 row)

gpdw=# insert into test001 select generate_series(1,1000000);
INSERT 0 1000000
gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 152 MB
(1 row)
```

​	通过上面案例可以说明一个问题就是vacuum并不能释放数据空间,其实是特别简单，vacuum 只是整理数据页的数据,将删除的数据剔除掉,之后将没有删除的数据按照新顺序保留在这个数据页面上。不过经常的vacuum可以释放这个文件空间可以放入新数据，从而不会导致数据膨胀。

​	

### case two:update  vacuum full



```

gpdw=# drop table test001;
DROP TABLE
gpdw=# create table test001(id int4);
NOTICE:  Table doesn't have 'DISTRIBUTED BY' clause -- Using column named 'id' as the Greenplum Database data distribution key for this table.
HINT:  The 'DISTRIBUTED BY' clause determines the distribution of data. Make sure column(s) chosen are the optimal data distribution key to minimize skew.
CREATE TABLE
gpdw=# insert into test001 select generate_series(1,1000000);
INSERT 0 1000000
gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 117 MB
(1 row)

gpdw=#  update test001 set id = id + 100;
UPDATE 1000000
gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 151 MB
(1 row)

gpdw=# select oid,relname from pg_class where relname='test001';
  oid  | relname 
-------+---------
 24595 | test001
(1 row)

gpdw=# vacuum full test001;
VACUUM
gpdw=# select oid,relname from pg_class where relname='test001';
  oid  | relname 
-------+---------
 24595 | test001
(1 row)

gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 117 MB
(1 row)
```

​	通过上面这个案例，可以发现vacuum full重新整理表test001,这反而是我们不建议操作的命令，vacuum full整理的过程通俗的理解是create table as,但是当表比较大时，它的消耗Io就非常高，然而有个现实情况，如果中间vauum full table中间出错时，可能导致表就不存在了（他其实是在这个数据文件上）



### case three :update truncate



```
--A
gpdw=# create table test001(id int4);
NOTICE:  Table doesn't have 'DISTRIBUTED BY' clause -- Using column named 'id' as the Greenplum Database data distribution key for this table.
HINT:  The 'DISTRIBUTED BY' clause determines the distribution of data. Make sure column(s) chosen are the optimal data distribution key to minimize skew.
CREATE TABLE
gpdw=# insert into test001 select generate_series(1,1000000);
INSERT 0 1000000
gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 117 MB
(1 row)

gpdw=# begin;
BEGIN
gpdw=# select oid,relname from pg_class where relname ='test001';
  oid  | relname 
-------+---------
 24601 | test001
(1 row)

gpdw=#  update test001 set id = id + 100;
UPDATE 1000000
gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 151 MB
(1 row)

gpdw=# truncate table test001;
TRUNCATE TABLE
gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 151 MB
(1 row)

```

  在事物中发现数据库进行truncate也并没有发生数据空间变化，那么查看一下事物内和另外一个事物中,表的oid和文件路径是否发生变化

```
--A

gpdw=# select oid,relname from pg_class where relname ='test001';
  oid  | relname 
-------+---------
 24601 | test001
(1 row)

gpdw=# select pg_relation_filepath('test001');
 pg_relation_filepath 
----------------------
 base/16384/24585
(1 row)

```

​	新开启的事物B

```
--B
gpdw=# select oid,relname from pg_class where relname ='test001';
  oid  | relname 
-------+---------
 24601 | test001
(1 row)

gpdw=# select pg_size_pretty(pg_database_size('gpdw'));
 pg_size_pretty 
----------------
 151 MB
(1 row)

gpdw=# select pg_relation_filepath('test001');
 pg_relation_filepath 
----------------------
 base/16384/24584
```

 发现除了pg_relation_filepath不一样外，其他保持一样(在这里标注一下,pg都是发生变化的)



```
--事物A
gpdw=# end;
COMMIT
gpdw=# select pg_relation_filepath('test001');
 pg_relation_filepath 
----------------------
 base/16384/24585
(1 row)
```

```
--事物B
gpdw=# select pg_relation_filepath('test001');
 pg_relation_filepath 
----------------------
 base/16384/24585
(1 row)

gpdw=# select oid,relname from pg_class where relname ='test001';
  oid  | relname 
-------+---------
 24601 | test001
(1 row)
```

​	事物A提交后，事物B的路径是否发生变化;通过这个案例可以说明这个问题，truncate可以释放空间,可以被其他表使用，因此后期中间表建议可以如下操作`truncate table xxx;update xxx;truncate table xxx;`这个流程，这样就不会出现数据空间冗余浪费的情况。



## summary

​	因为pg的mvcc的update机制,加上很少会开启自动autovacuum导致gp的膨胀率非常高.不过在现实中要避免这个情况尤其是update时比较困难的,在函数中添加vacuum是不被允许的,通过人工来做这件事情,这是很难办到的事情,因此就要经常对表进行analyze操作(当然建议在晚上做),通过脚本发现`select * from gp_toolkit.gp_bloat_diag;select (btdrelpages-btdexppages)*8/1024/1024 as GB,* from gp_toolkit.gp_bloat_expected_pages where btdrelpages-btdexppages >0; `,利用psql -c 命令帮助做成这件事情就可以了

```
$ psql -d gpdw -U gpadmin -c "vacuum test001";
```

或者使用psql -f命令帮助执行

```
$ vim psql_vacuum.txt 
vacuum test001;
vacuum test002;
$ psql -d gpdw -U gpadmin -f ./psql_vacuum.txt 
VACUUM
VACUUM
```
