## 4.1 数值类型

​	数值类型包括整数类型，任意精度数字类型，浮点类型，序列类型，货币类型等。



### 4.1.1 整数类型

​	smallint,integer,bigint都是整数类型，每种类型只能存储一定范围的整数，超过该范围就会报错。

| 名字     | 存储容量 | 说明           | 范围                                     |
| -------- | -------- | -------------- | ---------------------------------------- |
| smallint | 2字节    | 小范围整数     | -32768~32767                             |
| integer  | 4字节    | 整数的常见选择 | -2147483648~2147483647                   |
| bigint   | 8字节    | 大范围整数     | -9223372036854775808~9223372036854775807 |

​	常见的整数类型integer，因为它提供了在范围，存储空间和性能之间的最佳平衡

​	一般只有在磁盘容量不足的时候才使用smallint。

​	而只有integer类型的范围不够时才能bigint类型

【举例】

```
ysys=# CREATE TABLE test_integer(id1 integer,id2 smallint,id3 bigint);
CREATE TABLE
ysys=# INSERT INTO test_integer VALUES(32400,1024,2147483648);
INSERT 0 1
ysys=# select * from test_integer ;
  id1  | id2  |    id3     
-------+------+------------
 32400 | 1024 | 2147483648
(1 row)
```



### 4.1.2 任意精度数字类型

​	任意精度数字类型包括numeric和decimal。两者在使用上是等价的



| 名字    | 存储容量 | 说明               | 范围                                      |
| ------- | -------- | ------------------ | ----------------------------------------- |
| numeric | 可变     | 用户指定精度，精确 | 最高小数点前131072位，以及小数点后16383位 |
| decimal | 可变     | 用户指定精度，精确 | 最高小数点前131072位，以及小数点后16383位 |

​	numeric类型通过numeric(precision,scale)来定义

- precision称为精度，表示总共的位数

- scale称为比例，表示小数点部分的位数

  如果用户插入数据小数位数超过scale，则系统自动会进行四舍五入，如果用户插入的数据超出最大范围，则插入失败

【举例】

```
ysys=# CREATE TABLE test_num(id numeric(6,2),id2 decimal(4,0));
CREATE TABLE
ysys=# INSERT INTO test_num VALUES(1024.44,2.3);
INSERT 0 1
ysys=# SELECT * FROM test_num ;
   id    | id2 
---------+-----
 1024.44 |   2
(1 row)
```



### 4.1.3 浮点类型

​	PostgreSQL中的浮点类型用来存储小数，浮点类型包括real和double precision。

| 名字             | 存储容量 | 说明             | 范围           |
| ---------------- | -------- | ---------------- | -------------- |
| real             | 4 字节   | 可变精度，不精确 | 6位十进制精度  |
| double precision | 8 字节   | 可变精度，不精确 | 15位十进制精度 |

​	在大部分平台中，real类型范围是-1E+37~1E+37,精度至少是6位数字，double precision类型的范围是-1E+308~1E+308,精度至少是15位精度，太大或者太小都会导致报错

​	PostgreSQL还支持SQL标准表示法float和float(p),用于声明非精确的数字类型。在这里，p指定以二进制位表示的最低可接受精度。

​	real和double precision都是不准确的，可变精度的数字类习惯，这意味着一些值不能精确地转换成内部格式，而是以近似的形式存储的。因此，存储和检索一个值可能出现一些缺失。如果要进行准确的数据存储和计算，则应该使用numeric类型。

【举例】

```
ysys=# CREATE TABLE test_fp(id1 real,id2 double precision,id3 float(8));
CREATE TABLE
ysys=# INSERT INTO test_fp VALUES(1024,2.3,4.72345789);
INSERT 0 1
ysys=# INSERT INTO test_fp VALUES(1024.4455,2.3,4.7);
INSERT 0 1
ysys=# SELECT * FROM test_fp ;
   id1   | id2 |   id3   
---------+-----+---------
    1024 | 2.3 | 4.72346
 1024.45 | 2.3 |     4.7
```

​	在PostgreSQL中，浮点类型有3个特殊值，即Infinity,-Infinity,NaN，分别代表正无穷大，负无穷大，非数字





### 4.1.4 序列类型

​	smallserial,serial和bigserial不是真正的数据类型，它们作为唯一标识符列而存在，用户在需要使用自增整数时，可以选用这三者之一。如果一个表中本身的字段中，没有诗意作为主键的，则可以增加一个专门的字段并指定serial类型作为主键。

| 名字        | 存储容量 | 说明             | 范围                  |
| ----------- | -------- | ---------------- | --------------------- |
| smallserial | 2 字节   | 自动增加的小整数 | 1~32767               |
| serial      | 4 字节   | 自动增加的整数   | 1~2147483647          |
| bigserial   | 8 字节   | 自动增加的大整数 | 1~9223372036854775807 |

【举例】

```
ysys=# CREATE TABLE test_seq(id1 SERIAL,num INTEGER,id3 FLOAT(8));
CREATE TABLE
ysys=# INSERT INTO test_seq(num,id3) VALUES(3,4.7);
INSERT 0 1
ysys=# INSERT INTO test_seq(num,id3) VALUES(4,4.7);
INSERT 0 1
ysys=# SELECT * FROM test_seq;
 id1 | num | id3 
-----+-----+-----
   1 |   3 | 4.7
   2 |   4 | 4.7
(2 rows)
```



### 4.1.5 货币类型

​	money类型用于存储固定小数精度的货币数字，其小数的精度由数据库系统的lc_monetary参数设置决定。lc_monetary参数用来指定货币符号对应的地区，与时区概念类似

​	money类型可以接受很多输入格式，包括整数和浮点数，以及常用的货币格式,如“$1,000.00".输出通常是货币格式。不同区域的货币符号有所不同。

| 名字  | 存储容量 | 说明   | 范围                                           |
| ----- | -------- | ------ | ---------------------------------------------- |
| money | 8字节    | 货币额 | -9223372036854775808.08~9223372036854775807.07 |

​	lc_monetary参数的默认设置是"en_US.utf8"，其货币格式美元，如果要输出为人民币格式，则需要将lc_moneraty参数设置为`zh_CN.utf8`

```
ysys=# SELECT '52093.89'::money;
   money    
------------
 $52,093.89
(1 row)

ysys=# show lc_monetary;
 lc_monetary 
-------------
 en_US.UTF-8
(1 row)

ysys=# set lc_monetary='zh_CN.utf8';
SET
ysys=# SELECT '52093.89'::money;
    money    
-------------
 ￥52,093.89
(1 row)
```

## 4.2 字符串类型



### 4.2.1 字符串类型分类

​	PostgreSQL提供了三种基础的字符串类型:character(char),character varying(varchar),text

| 名字       | 说明                                 |
| ---------- | ------------------------------------ |
| char(n)    | 固定长度字符串，长度不足时以空格补充 |
| varchar(n) | 有长度限制的变长字符串               |
| text       | 无长度限制的变长字符串               |

​	character(n)占用的存储空间是固定的，其中正整数n表示最多存储n个字符

- 如果输入的字符串长度超过n，则PostgreSQL会抛出报错。
- 如果字符串长度比n短，则数据库会自动使用空格填满，不管存储还是查询显示时都是如此。

   character varyning(n)和text都使用与存储长度可变的文本，根据每行记录的长度分配相应的空间，两者主要区别在于长度n的限制。对于character varyning(n)来说，正整数n同样表示最多存储n个字符的字符串。

【举例】

```
ysys=# CREATE TABLE test_char(a character(4),b varchar(5));
CREATE TABLE
ysys=# INSERT INTO test_char VALUES('ok','test');
INSERT 0 1
ysys=# SELECT * FROM test_char;
  a   |  b   
------+------
 ok   | test
(1 row)
```



### 4.2.2 使用连接运算符

​	连接运算符可以作用所用字符串类型，对于字符串类型来说，自动填充的空格对连接没有任何影响，连接时会被自动去掉。

| 运算符                                      | 返回类型 | 说明                       | 示例            | 结果     |
| ------------------------------------------- | -------- | -------------------------- | --------------- | -------- |
| string\|\|string                            | text     | 串接                       | ‘Post'\|\|'gre' | Postgre  |
| string\|\|non-string 或non-string\|\|string | text     | 使用一个非字符串输入的串接 | 'Value:'\|\|42  | Value:42 |

【举例】

```
ysys=# SELECT a,b,b||a FROM test_char;
  a   |  b   | ?column? 
------+------+----------
 ok   | test | testok
(1 row)

ysys=# SELECT a,a||4 FROM test_char;
  a   | ?column? 
------+----------
 ok   | ok4
(1 row)
```



### 4.2.3 使用模式匹配运算符

​	PostgreSQL提供了3种独立实现模式匹配的方法:LIKE 运算符,SIMILAR TO 运算符,POSIX 运算符

**1.LIKE 运算符**

​	LIKE运算符语法格式如下

```
string LIKE pattern [ESCAPE escape-character]
string NOT LIKE patttern [ESCAPE escape-character]
```

【举例】

```
ysys=# SELECT 'ABC' LIKE 'ABC';
 ?column? 
----------
 t
(1 row)

ysys=# SELECT 'ABC' NOT LIKE 'ABCD';
 ?column? 
----------
 t
(1 row)
```

​	LIKE运算符总是覆盖整个字符串。因此，要匹配在字符串中任何位置的序列，该模式必须以百分号开头或者结尾，在patttern中的下划线表示匹配的任何单个字符，百分号表示匹配0个或者多个字符的序列.

​	如果pattern不包含百分号或下划线，则改模式只表示它本身的字符串，这是，LIKE运算符的作用相当于等号运算符

​	为了匹配`_`和`%`字符本身，在相应的模式匹配字符串中，这两个字符必须冠以转义字符。

​	默认的转义字符是反斜线`\`,如果想使用其他转义字符，则需要使用ESCAPE子句进行声明。如果想匹配转义字符本身，则连续写两个转义字符。

【举例】

```
               ^
ysys=# SELECT 'abc_de%' like '%\_%';
 ?column? 
----------
 t
(1 row)

ysys=# SELECT 'abcde%fg' like '%\%%';
 ?column? 
----------
 t
(1 row)

ysys=# SELECT 'abcde\fg' like '%\\%' ESCAPE '|';
 ?column? 
----------
 f
(1 row)
```



**2.SIMILAR TO 运算符**

​	SIMILAR TO 运算符的语法格式如下

```
string SIMILAR TO pattern [ESCAPE escape-character]
string NOT SIMILAR TO pattern [ESCAPE escape-character]
```

​	SIMILAR TO 运算符根据自己的模式是否匹配字符串，而返回TRUE或者FALSE。

​	SIMILAR TO 运算符和LIKE运算符非常相似，只不是它是使用了SQL标准定义的正则表达式模式。

​	SIMILAR TO 运算符只能在它的模式匹配整个字符串时才能成功，这一点和普通正则表达式的行为不同

​	在普通的正则表达式中，模式匹配字符串的任意部分。和Like运算符类似的地方还有,SIMILAR TO运算符分别使用`_`和`%`表示任意单个字符和任意字符串的通配符。



​	SIMILAR TO运算符支持POSIX正则表达式的模式匹配元字符。

```
|:表示选择
*:表示重复前面的项零次或者更多次
+:表示重复前面的项一次或者更多次
?:表示重复前面的项零次或一次
{m}:表示重复前面的项m次
{m,}:表示重复前面的项m次或更多次
{m,n}:表示重复前面的项至少m次，且不超过n次
```

​	可以使用圆括号()将多个项组合成一个逻辑项。

​	一个方括号表达式[...]声明一个字符类，就像POSIX正则表达式一样。

【举例】

```
postgres=# SELECT 'abc' SIMILAR TO 'a';
 ?column? 
----------
 f
(1 row)

postgres=# SELECT 'abc' SIMILAR TO 'abc';
 ?column? 
----------
 t
(1 row)

postgres=# SELECT 'abc' SIMILAR TO '%(b|d)%';
 ?column? 
----------
 t
(1 row)

postgres=# SELECT 'abc' SIMILAR TO '(b|c)%';
 ?column? 
----------
 f
(1 row)


```



**3.POSIX正则表达式**

| 运算符 | 说明                           |
| ------ | ------------------------------ |
| ~      | 匹配正则表达式，区分大小写     |
| ~*     | 匹配正则表达式，不区分大小写   |
| !~     | 不匹配正则表达式，区分大小写   |
| !~*    | 不匹配正则表达式，不区分大小写 |

【举例】

```
postgres=# SELECT 'abc' ~ 'abc';
 ?column? 
----------
 t
(1 row)

postgres=# SELECT 'abc' ~ '^a';
 ?column? 
----------
 t
(1 row)

postgres=# SELECT 'abc' ~ '(b|d)';
 ?column? 
----------
 t
(1 row)

postgres=# SELECT 'abc' ~ '^(b|d)';
 ?column? 
----------
 f
(1 row)
```





## 4.3 二进制数据类型

​	PostgreSQL提供了二进制(bytea)数据类型用于存储二进制字符串，该数据类型占用空间容量为1字节或者4字节加上实际的二进制字符串



### 4.3.1 二进制数据类型简介

​	二进制数据类型可以存储不可见的字符，而字符串类型则不能存储不可见的字符。

​	

- 二进制数据类型适用于存储程序员认为是`裸字节`的数据
- 字符串适用于存储文本。



### 4.3.2 二进制数据的十六进制格式

​		

​	十六进制格式将二进制数据编码为字符串，每个字节表示两个十六进制数，最高有效位在前。

​	十六进制字符串以`\x`开头

​	【举例】

```
SELECT E'\\xDEADBEEF'
```

​		十六进制格式可以是大写，也可以是小写，在十六进制格式之间可以有空白

​		十六进制格式和很多外部应用兼容，兵器在转换速度要比转义格式更快，因此人们更愿意使用十六进制格式。



### 4.3.3 二进制数据的转义格式

​	转义格式是二进制数据类型的传统PostgreSQL格式，它将采用将二进制数据表示成ASCII字符序列的方法，而将那些无法用ASCII字符表示的字节转换成特殊的转义语句。

​	要转义一个字节数值，通常需要将它的数值转换成赌赢的三位八进制数，并且加两个前导反斜线（`\\`）,有的字符可以加一个反斜线直接转义，如单引号和反斜线本身。

| 十进制字节数值 | 说明             | 转义输入表示          | 示例                     | 结果   |
| -------------- | ---------------- | --------------------- | ------------------------ | ------ |
| 0              | 0字节            | `E'\\000'`            | `SELECT E'\\000'::bytea` | `\000` |
| 39             | 单引号           | ''''或`E'\\047'`      | `SELECT E'\'::bytea`     | '      |
| 92             | 反斜线           | `E'\\\\'`或`E'\\134'` | `SELECT E'\\\\'::bytea`  | `\\`   |
| 0~31和127~255  | “不可打印的”字节 | `E'\\xxx'`            | `SELECT E'\\001'::bytea` | `\001` |



## 4.4 日期类型和时间类型



### 4.4.1 日期类型和时间类型分类

| 名字                                | 存储容量 | 说明                             | 最小值       | 最大值        | 分辨率     |
| ----------------------------------- | -------- | -------------------------------- | ------------ | ------------- | ---------- |
| `timestamp[(p)][without time zone]` | 8字节    | 包括日期和时间(无时区)           | 4713BC       | 294296AD      | 1毫秒/14位 |
| `timestamp[(p)] with time zone`     | 8字节    | 包括日期和时间(有时区)           | 4713BC       | 294296AD      | 1毫秒/14位 |
| date                                | 4字节    | 日期(没有一天中的时间)           | 4713BC       | 5874897AD     | 1天        |
| `time[(p)][without time zone]`      | 8字节    | 一天中的时间(没有日期)           | 0:00:00      | 24:00:00      | 1毫秒/14位 |
| `time[(p)][with time zone]`         | 12字节   | 一天中的时间(没有日期，带有时区) | 0:00:00+1459 | 24:00:00-1459 | 1毫秒/14位 |
| inteval [fields] [(p)]              | 16 字节  | 时间间隔                         | 负很大       | 正很大        | 1毫秒/14位 |

​	time,timestamp,interval接受一个可选的精度p，这个精度值声明在秒域中小数点之后爆率的位数。

​	在默认情况下，在精度上没有明确的边界,p允许的范围是0-6

​	interval有一个附件选项，他可以通过下面选项限制存储领域的集合

`YEAR,MONTH,HOUR,MINUTE,SECOND,YEAR TO MONTH,DAY TO HOUR,DAY TO MINUTE,DAY TO SECOND,HOUR TO MINUTE,HOUR TO SECOND,MINUTE TO SECOND`

​	

### 4.4.2 日期类型和时间类型输入格式

​	日期类型和时间类型的输入格式包括ISO 8601,SQL,Postgres.

**1.日期类型输入**

为了避免混淆年月日的顺序，可以通过配置datestyle参数指定顺序。

- 设置为MDY
- 设置为DMY
- 设置为YMD

日期类型的输入格式

| 示例           | 说明                                                     |
| -------------- | -------------------------------------------------------- |
| 1999/1/8       | ISO 8661 格式下，任何模式下的1999年1月8日                |
| January 8,1999 | 在任何模式下都无歧义                                     |
| 1/8/1991       | MDY 1991年1月8日;DMY 1991年8月1日                        |
| 1/18/1991      | MDY 1991年1月18日；在其他模式下被拒绝                    |
| 2001/2/3       | MDY 2003年1月2日 DMY 2003年2月1日(WHAT),YMD 2001年2月3日 |
| 1999-Jan-08    | 任何模式下都是1991年1月8日                               |
| Jan-08-1999    | 任何模式下都是1991年1月8日                               |
| 8-Jan-1991     | 任何模式下都是1991年1月8日                               |

| 示例            | 说明                                  |
| --------------- | ------------------------------------- |
| 99-Jan-08       | YMD模式下的1999年1月8日，否则错误     |
| 8-Jan-99        | 1999年1月8日，除了在YMD模式下是错误的 |
| Jan-08-99       | 1991年1月8日，除了在YMD模式下是错误的 |
| 19910108        | ISO 8601格式，任何模式下1999年1月8日  |
| 990108          | ISO 8601格式，任何模式下1999年1月8日  |
| 1999.008        | 1999年的第八天，也就是1999年1月8日    |
| J2451187        | 儒略日期                              |
| January 8,99 BC | 公元前99年                            |



**2.时间类型输入**

​	当时间类型time[(p)] without time zone 和 time[(p)] with time zone.只写time,等同于time without time zone.



| 示例                                 | 说明           |
| ------------------------------------ | -------------- |
| 04:05:06:789                         | ISO 8601       |
| 4:05:06                              | ISO 8601       |
| 5:05                                 | ISO 8601       |
| 40506                                | ISO 8601       |
| 4:05 AM                              | 和04:05一致    |

### 4.4.4 时区

​	建议在使用时区时，使用同时包含日期和时间的数据类型。

​	在系统内部，所有与时区相关的日期和时间都是用UTC存储的。日期在客户端输出之后会被转换成由timezone参数指定的本地时间。

​	Postgresql可以使用3种形式不同指定分区

​	

**1.一个完整的时区名字**

​	例如，America/New_York。能够被识别的时区名字列在pg_timezone_names视图中。

​	Postgresql采用了IANA时区数据实现该目的，因此，相同的时区名字也可以在其他软件中被识别。

​	

**2.一个时区缩写**

​	例如，PST.这样一种声明仅仅定义了到UTC的一个特定偏移，而不像完整时区名那种指出整套夏令时转换日期规则。能够被识别的缩写列在pg_timezone_abbrevs视图中。



**3.POSIX时区声明**

​	POSIX时区声明的形式为STDoffset或STDoffsetDST

- STD是一个区域缩写。
- offset是本地时间相对于UTC时间的偏移量，该偏移量以小时作为
- DST是一个可选的夏令时区缩写



​	一个时区缩写可以是一个字母的字符串，或者由尖括号(<>)包围的任意字符串。当时区中包含夏令时区缩写时，PostgreSQL会根据IANA时区数据库的夏令时转换规则来使用该缩写。

## 4.6 位串类型

### 4.6.1 位串类型简介

​	位串类型是由1和0组成的字符串，该类型主要用于存储和可视化位掩码。

​	PostgreSQL支持两种SQL标准位串类型:bit(n)和bit varying(n),n是一个正整数

- bit类型的数据必须准确匹配长度n.如果用户输入短些或者长些的位串，则PostgreSQL会提示错误。如果不带长度n,则bit等效于bit(1)
- bit varying类型用于存储长度为n的位串类型，存储长度超过n的位串则会被拒绝。没有长度n的bit varying意味这没有长度限制。

【举例】

```
ysys=# create table test_bit(a bit(3),b bit varying(5));
CREATE TABLE
ysys=# insert into test_bit values(B'101',B'00');
INSERT 0 1
ysys=# insert into test_bit values(B'10',B'101');
ERROR:  bit string length 2 does not match type bit(3)
ysys=# insert into test_bit values(B'10'::bit,B'101');
ERROR:  bit string length 1 does not match type bit(3)
ysys=# insert into test_bit values(B'10'::bit(3),B'101');
INSERT 0 1
ysys=# select * from test_bit;
  a  |  b  
-----+-----
 101 | 00
 100 | 101
```



### 4.6.2 使用位串类型运算符

​	需要注意的是,`&`,`|`,`#`的位串操作数必须等长。位串在移位时会保留原始的长度。

| 运算符 | 说明     | 示例               | 结果     |
| ------ | -------- | ------------------ | -------- |
| \|\|   | 连接     | B'10001'\|\|B'011' | 10001011 |

| \|     | 按位或   | B'10001'\|B'01101' | 11101    |
| #      | 按位异或 | B'10001'#B'01101'  | 11100    |
| -      | 按位求反 | ~B'10001'          | 1110     |

## 4.9 JSON数据类型

### 4.9.1 JSON数据类型简介

​	根据RFC 7159中的说明,JSON数据类型用来存储JSON(javascript Object Notation)数据的。JSON是一种轻量级的数据交换格式，它基于ECMAScript的一个子集。

​	JSON数据类型采用完全独立的语言文本格式，但是也使用了类似于C语言的习惯。这些特性使JSON成为理想的数据交换格式，它容易被人阅读和编写，同时也容易被机器解析和生成。

​	定义为JSON数据类型能够强制要求每个被存储的值符号JSON规则，并且可以使用JSON相关的函数和运算符

​	PostgreSQL有以下两种JSON数据类型，两者输入数据的规则几乎相同，主要区别是效率。

- JSON:存储输入文本的精确复制，处理函数在每次执行时必须重新解析该数据。
- JSONB:这种类型的数据被存在一种分解好的二进制格式中，当输入这种类型的数据要稍微慢一些，因此需要进行附加转换，但是jsonb类型数据的处理速度要快很多，因为不需要解析，此外，jsonb类型数据也支持索引，这也是优势之一。



​	Json数据类型输入格式有一些限制，这些限制本身并不适用于对应的Postgresql数据类型，如所示

| JSON数据类型 | PostgreSQL数据类型 | 说明                                                         |
| ------------ | ------------------ | ------------------------------------------------------------ |
| string       | text               | 如果数据库编码不是UTF8,则作为非ASCIIUnicode转义符，“\u0000”是不允许的 |
| number       | numeric            | 不允许是NaN值和infinity值                                    |
| boolean      | boolean            | 只接受小写true和false                                        |
| null         | (无)               | SQL中的null表示不同含义                                      |



### 4.9.2 JSON 数据类型的输入/输出语法

​	RFC7159定义了JSON数据类型的输入、输出语法。JSON数据类型支持简单的基本值，如数值类型，字符串类型，布尔类型和null类型等

```
postgres=# select '"aa"'::json;
 json 
------
 "aa"
(1 row)
postgres=# select '5'::jsonb;
 jsonb 
-------
 5
(1 row)
```

​	JSON数据类型支持较多元素的数组，元素不需要为同一类型

```
postgres=# select '[1,2,"foo",null]'::json;
       json       
------------------
 [1,2,"foo",null]
(1 row)

```

​	JSON数据类型支持包含键值的对象。需要注意的是，包含键值的对象必须是带引号的字符串

```
postgres=# select '{"bar":"baz","balance":7.77,"active":false}'::jsonb;
                      jsonb                       
--------------------------------------------------
 {"bar": "baz", "active": false, "balance": 7.77}
(1 row)

postgres=# select '{"bar":"baz","balance":7.77,"active":false}'::json;
                    json                     
---------------------------------------------
 {"bar":"baz","balance":7.77,"active":false}
(1 row)
```

​	数组和对象可以被任意嵌套

```
postgres=# select '{"foo":[true,"bar"],"tags":{"a":1,"b":null}}'::json;
                     json                     
----------------------------------------------
 {"foo":[true,"bar"],"tags":{"a":1,"b":null}}
(1 row)

postgres=# select '{"foo":[true,"bar"],"tags":{"a":1,"b":null}}'::jsonb;
                        jsonb                        
-----------------------------------------------------
 {"foo": [true, "bar"], "tags": {"a": 1, "b": null}}
(1 row)

```



### 4.9.3 使用json运算符

​	json类型和jsonb类型都支持的运算符

| 运算符 | 右操作数类型 | 说明                                      | 示例 | 结果 |
| ------ | ------------ | ----------------------------------------- | ---- | ---- |
| ->     | int          | 获取JSON数组元素(索引从0开始，负整数结束) |      |      |
| ->     | text         | 通过键值获取JSON对象域                    |      |      |
| ->>    | int          | 以文本形式获取JSON数组元素                |      |      |
| ->>    | text         | 以文本形式获取JSON对象域                  |      |      |
| #>     | text[]       | 获取指定路径的JSON对象                    |      |      |
| #>>    | text[]       | 以文本形式获取在指定路径的JSON对象        |      |      |
